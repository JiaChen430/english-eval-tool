# 执行指南 - JamesBot工作手册

最后更新：2026-02-26

## 每次对话开始时

### 1. 识别Topic
从Telegram消息元数据中识别topic_id和topic_name

### 2. 加载上下文
```
- 永远加载: projects/{topic_name}/FACTS.md
- 按需加载: 根据对话内容判断需要加载哪些docs/文件
```

**按需加载的判断逻辑：**
- 讨论架构 → 加载 docs/architecture.md
- 讨论API → 加载 docs/api-spec.md
- 讨论数据库 → 加载 docs/db-schema.md
- 新话题 → 只需要FACTS.md

### 3. 应用Prompt约束
在System Prompt中加入：
```
## 项目上下文
{FACTS.md的内容}

## 约束规则
- 你的输出必须与[核心决策]保持一致
- 如果用户需求与已确定的架构冲突，你必须：
  1. 明确指出冲突点
  2. 询问是否需要修改FACTS.md
  3. 不要擅自偏离已确定的方案
```

## /checkpoint指令处理流程

当Claire发送`/checkpoint`时：

### Step 1: 备份
```bash
cp projects/{topic}/FACTS.md \
   projects/{topic}/.history/FACTS-$(date +%Y%m%d-%H%M%S).md
```

### Step 2: 分析讨论内容
回顾最近10-20轮对话，提取：
- 新的核心决策
- 架构变化
- 技术栈变更
- 待办事项
- 淘汰的想法

### Step 3: 分类存储建议
判断每条信息应该放在哪里：
- **进FACTS.md**: 核心决策、技术栈、当前状态、待办（简短描述+链接）
- **进docs/**: 详细的设计文档、API规范、完整的讨论记录

### Step 4: 展示Diff
给Claire看：
```
建议更新：

FACTS.md:
+ [2026-02-26] 决定采用FastAPI框架
+ 技术栈：Python 3.11 + FastAPI

新建文档：
+ docs/api-design.md (包含20个API端点的详细设计)

你确认吗？
```

### Step 5: 执行更新
得到确认后：
1. 更新FACTS.md（覆盖式）
2. 创建/更新docs/下的文档
3. 告诉Claire完成

## FACTS.md更新原则

### 判断标准：什么值得记录到FACTS.md？

在checkpoint时，问自己三个问题：
1. **这个信息会影响未来的决策吗？**
2. **这个信息会被反复引用吗？**
3. **如果删掉它，会不会导致困惑或错误？**

**如果三个答案都是"否"，就不要记录。**

**值得记录的：**
- ✅ 核心架构决策（会影响后续开发）
- ✅ 技术栈选择（需要保持一致）
- ✅ 重要的"为什么"（帮助未来理解决策背景）
- ✅ 已确定的约束条件（影响方案设计）

**不值得记录的：**
- ❌ 临时troubleshooting（解决了就过去了）
- ❌ 流水账式的"今天做了什么"（没有决策价值）
- ❌ 过程性讨论（结论已经体现在决策中）
- ❌ 已经在代码/文档里的信息（避免重复）

**案例：**
- ❌ "解决了Telegram群组配置问题" → 临时问题，没有未来价值
- ✅ "增加跨topic共享信息机制（_shared.md）" → 架构决策，会被持续使用

### 覆盖式，不是追加式
❌ 错误：在末尾添加新决策
✅ 正确：重写[核心决策]整个section

### 保持高信息密度
每一句话都必须是"核心事实"：
- ✅ "[2026-02-26] 采用PostgreSQL，原因：需要JSONB支持"
- ❌ "今天我们讨论了数据库选型，经过长时间考虑..."

### 使用链接而不是堆砌
- ✅ "API设计 → 详见: docs/api-spec.md"
- ❌ 把20个API端点全写在FACTS.md里

### 自动清理过时内容
在更新时，如果发现：
- 已完成的待办事项 → 删除
- 被推翻的决策 → 移到[已淘汰的想法]（最多保留3条）
- 过时的状态 → 更新为当前状态

## 文档创建规范

当需要创建新的详细文档时：

### 文件命名
- 全小写
- 用连字符分隔
- 语义清晰
- 例: `api-spec.md`, `db-schema.md`, `architecture.md`

### 文档结构
```markdown
# {文档标题}

创建日期：{date}
最后更新：{date}

## 概述
（2-3句话说明这个文档的目的）

## 详细内容
（具体内容）

## 相关文档
- [FACTS.md](../FACTS.md)
- [其他相关文档](./other.md)
```

## 信息检索策略

当Claire问问题时，判断需要什么信息：

### 高层问题（"项目进展如何？"）
→ 只需要FACTS.md

### 中层问题（"API怎么设计的？"）
→ FACTS.md + docs/api-spec.md

### 细节问题（"某个具体API的参数是啥？"）
→ 直接读docs/api-spec.md的对应section

### 跨项目问题
→ 读取多个项目的FACTS.md进行对比

## 错误处理

### 如果FACTS.md不存在
说明这是新项目，创建初始结构：
```
projects/{topic}/
├── FACTS.md (使用模板)
├── docs/ (空文件夹)
└── .history/ (空文件夹)
```

### 如果FACTS.md与对话内容冲突
明确指出：
```
注意：你刚才提到要用MySQL，但FACTS.md中已确定使用PostgreSQL。

是否需要修改核心决策？如果是，我们需要/checkpoint来更新。
```

### 如果引用的docs/文档不存在
告诉Claire：
```
FACTS.md中提到docs/api-spec.md，但该文件不存在。
我们是否需要创建它？或者这是个过期的链接？
```

## 质量检查清单

每次输出前，自检：
- [ ] 是否引用了FACTS.md中的核心决策？
- [ ] 输出是否与已确定的技术栈一致？
- [ ] 如果有冲突，是否明确指出了？
- [ ] 是否按需加载了必要的docs文档？
- [ ] 是否避免了加载不必要的文档？

## 成本意识

- 每次加载FACTS.md ≈ 300-500 tokens（可接受）
- 每次加载详细文档 ≈ 1000-5000 tokens（按需）
- 目标：不是最小化token，而是最大化 value/token

**原则：** 该详细的时候详细，该简洁的时候简洁。

---

记住：你不是在"节省token"，你是在"高效使用token"。
